# タスク: [機能名]

**入力**: 機能設計ドキュメント `/specs/[###-feature-name]/`  
**前提条件**: plan.md (必須)、spec.md (ユーザーストーリの場合必須)、research.md、data-model.md、contracts/

**テスト**: 以下の例にはテストタスクを含む。テストはオプション - 機能仕様で明確に要求された場合のみ含める。

**組織**: タスクをユーザーストーリごとにグループ化し、各ストーリーを独立して実装・テスト可能にする。

## 形式: `[ID] [P?] [Story] 説明`

- **[P]**: 並列実行可能。依存関係がない（異なるファイル、同一ジャーニー内の独立タスク）
- **[Story]**: 所属するユーザーストーリ、例：(US1, US2, US3)
- 各タスクに正確なファイルパスを記載

## パス規則

- **単一プロジェクト**: `src/`、`tests/` がリポジトリルート
- **Web アプリ**: `backend/src/`、`frontend/src/`
- **モバイル**: `api/src/`、`ios/src/` または `android/src/`
- 以下のパスは plan.md に基づいて調整

## 依存関係と実行順序

### フェーズの依存関係

- **セットアップ (フェーズ 1)**: 依存なし - すぐに開始可能
- **基盤 (フェーズ 2)**: セットアップ完了後 - 全ユーザーストーリーをブロック
- **ユーザーストーリ (フェーズ 3+)**: 基盤フェーズ完了後
  - 並列実行可能（チーム規模があれば）
  - または優先順位順（P1 → P2 → P3）
- **ポーランド (最終フェーズ)**: 必要なすべてのストーリーが完了後

### ユーザーストーリ間の依存関係

- **ユーザーストーリ 1 (P1)**: 基盤フェーズ後 - 他ストーリーへの依存なし
- **ユーザーストーリ 2 (P2)**: 基盤フェーズ後 - US1 と統合可能だが独立テスト可能
- **ユーザーストーリ 3 (P3)**: 基盤フェーズ後 - US1/US2 と統合可能だが独立テスト可能

### 各ストーリー内の依存関係

- テスト (含む場合) をまず作成し、失敗確認後に実装
- モデル → サービス → エンドポイント
- 実装 → 統合 → ポーランド

### 並列実行の機会

- セットアップ内 [P] タスクはすべて並列可能
- 基盤内 [P] タスクはすべて並列可能 (フェーズ 2 内)
- 基盤完了後、全ユーザーストーリーが並列開始可能 (チーム規模に応じて)
- ストーリー内 [P] タスクはすべて並列可能
- 異なるストーリーを異なるチームメンバーが並列作業可能
